Introduce a 'free list' type: lists where rows are custom labels (not team members). Moderators define rows, and any logged-in user can edit cells in public free lists. The cells table is reused: player_id stores the free_list_rows.id.

Purpose: Enable lists like training logs, match reports, or any event-based tracking where rows are not tied to individuals.
Output: New list_type column, free_list_rows table, full CRUD for free list rows and cells in the moderator UI.
</objective>

Steps:

   - Check `information_schema.columns` for `table_name='lists' AND column_name='list_type'`
   - If absent: `ALTER TABLE {$schema}.lists ADD COLUMN IF NOT EXISTS list_type VARCHAR(10) NOT NULL DEFAULT 'member' CHECK (list_type IN ('member', 'free'))`
   - Wrap in try/catch, log on error

2. Create `free_list_rows` table:
   - Check `information_schema.tables` for `table_name='free_list_rows'`
   - If absent: `CREATE TABLE {$schema}.free_list_rows (id SERIAL PRIMARY KEY, list_id INTEGER NOT NULL REFERENCES {$schema}.lists(id) ON DELETE CASCADE, label TEXT NOT NULL, position INTEGER NOT NULL DEFAULT 0, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW())`
   - Add index: `CREATE INDEX IF NOT EXISTS idx_flr_list_id ON {$schema}.free_list_rows(list_id)`
   - Wrap in try/catch, log on error

3. Add RLS on `free_list_rows`:
   - Always drop-and-recreate (idempotent): `DROP POLICY IF EXISTS flr_select ON ...` then `CREATE POLICY flr_select ... FOR SELECT USING (EXISTS (SELECT 1 FROM {$schema}.lists WHERE lists.id = free_list_rows.list_id AND (current_setting('app.is_admin',true)='true' OR (current_setting('app.current_role',true)='moderator' AND lists.team_id=...) OR (lists.visibility IN ('public','protected') AND lists.team_id=...))))`
   - `flr_insert`: `current_setting('app.current_role',true)='moderator' AND EXISTS (SELECT 1 FROM lists WHERE lists.id=free_list_rows.list_id AND lists.team_id=...)`
   - `flr_delete`: same as flr_insert
   - `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` on free_list_rows (guard with try/catch in case app user lacks superuser for FORCE; FORCE may fail on Hetzner shared — log and continue)
   - Also call `ALTER TABLE {$schema}.free_list_rows ENABLE ROW LEVEL SECURITY` without FORCE as fallback

**In `src/coach/list_detail_handler.php`:**

1. Fetch `list_type` in the list metadata query: `SELECT id, name, visibility, date, description, list_type FROM lists WHERE id = ?`
   - Guard: if `DB_HAS_LIST_TYPE` is false, treat list_type as 'member'.

2. Branch on `$list['list_type']`:
   - **For 'free' lists:**
     - Skip member query. Instead: `SELECT id, label, position FROM free_list_rows WHERE list_id = ? ORDER BY position, created_at`
     - Store result as `$free_rows`
     - Fetch columns (local only — no global columns): `SELECT c.id, c.name, c.data_type, c.coach_only FROM columns c WHERE c.is_active=TRUE AND c.list_id=? ORDER BY c.sort_order, c.created_at` — uses `$list_id`
     - Fetch cells: `SELECT player_id AS row_id, column_id, value FROM cells WHERE list_id = ?` — build `$cells[row_id][column_id]`
     - Set `$players = []` (empty, not used in free list template path)
   - **For 'member' lists:** existing code unchanged.

3. Handle POST for free list bulk save:
   - Submitted data: `$_POST['cells'][$row_id][$col_id]` — same shape as member list
   - Loop over `$free_rows` (not `$players`): for each row, for each column, upsert cell using `$row['id']` as `player_id`
   - Validation logic identical to member list (boolean/number/text)

4. Handle POST for row management (add row, delete row) — two new POST action values:
   - POST param `action=add_row`: `$label = trim($_POST['row_label'] ?? ''); $position = (int)($_POST['row_position'] ?? 0);` — validate label non-empty, insert into free_list_rows, redirect back
   - POST param `action=delete_row`: `$row_id = (int)($_POST['row_id'] ?? 0);` — verify row belongs to list (SELECT id FROM free_list_rows WHERE id=? AND list_id=?), then DELETE; also DELETE FROM cells WHERE list_id=? AND player_id=? (cleans orphan cells), redirect back
   - All POST actions require `require_csrf()` (already done at top of POST block)
   - Note: cells table `player_id` FK references users. For free list rows, we store free_list_rows.id here. This violates the FK if free_list_rows.id collides with users.id — but since both are SERIAL sequences from separate tables they may overlap. **Solution: do NOT delete cells via the FK cascade; instead manually `DELETE FROM cells WHERE list_id=? AND player_id=?` before deleting the free row. Alternatively, to avoid FK violation on INSERT, the cells.player_id FK must be relaxed.** 

   **IMPORTANT FK issue:** `cells.player_id` has `REFERENCES users(id)`. Inserting a free_list_rows.id that doesn't exist in `users` will fail. Fix in the migration (Task 1, add to Migration 006): `ALTER TABLE {$schema}.cells DROP CONSTRAINT IF EXISTS cells_player_id_fkey` — drop the FK on player_id. This is acceptable: the app layer enforces ownership; RLS enforces row visibility. Document the removal in a comment. Add this ALTER in Migration 006 before the free_list_rows table creation. If the ALTER fails (e.g. constraint name differs), use: `ALTER TABLE {$schema}.cells ALTER COLUMN player_id DROP NOT NULL` as a fallback approach — actually the correct approach is to just drop the FK:
   ```
   SELECT conname FROM information_schema.table_constraints
   WHERE table_schema='{$schema}' AND table_name='cells' AND constraint_type='FOREIGN KEY' 
   AND conname LIKE '%player_id%'
   ```
   then drop it. Simpler: just attempt `ALTER TABLE {$schema}.cells DROP CONSTRAINT IF EXISTS cells_player_id_fkey` — PostgreSQL supports this. Also update `db_init_schema()` to remove the FK from the cells CREATE TABLE.
